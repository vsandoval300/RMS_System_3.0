Camino 1

Alta Manual

El usuario registra el negocio uno a uno desde el admin panel. El ciclo completo DFT → PND → APR se recorre negocio por negocio. Las acciones están en las páginas Edit y View de cada Business.

Camino 2

Carga Masiva por Batch Import

El usuario sube un archivo Excel con múltiples negocios. Se crea un Import Batch (IMP-YYYY-NNNN) que agrupa todos los registros. El Manager aprueba o rechaza el batch completo. Sin ciclo por negocio individual.

Comportamiento del Reject · Cascade

Al rechazar un batch, se aplica soft-delete en orden inverso de dependencias:

  1. businessdoc_schemes
  2. businessdoc_insureds
  3. operative_docs
  4. liability_structures
  5. cost_nodesx
  6. cost_schemes
  7. businesses
Todos los registros tienen import_batch_id asignado al momento del import, lo que permite identificar y eliminar el grupo completo sin ambigüedad.

📬 Digest Semanal — PendingApprovalsDigestMail

Para evitar saturar la bandeja del Manager con correos individuales, todos los avisos al Manager (pending items) se consolidan en un solo correo semanal por aprobador.

  • Comando: rms:send-approvals-digest · Scheduled: lunes 8:00am · withoutOverlapping()
  • Destinatario: un correo por Manager, agrupando todo lo que le corresponde revisar
  • Contenido Camino 1: listado de businesses en status PND — código, descripción, creador, tiempo esperando
  • Contenido Camino 2: listado de Import Batches en status pending_review — código, total records, importer, tiempo esperando
  • Si el manager no tiene email: se omite silenciosamente (sin error)
  • Campanita: sigue siendo inmediata en ambos caminos — el digest solo aplica al canal mail

Jerarquía de Usuarios (manager_id) — Aplica a Ambos Caminos
La relación Manager → Creator/Importer es una auto-referencia en la tabla users:
users.id → users.manager_id
// self-referential FK, nullable
manager() → belongsTo(User, 'manager_id')
subordinates() → hasMany(User, 'manager_id')
Camino 1: el manager se resuelve como createdByUser->manager al momento de Submit for Review.
Camino 2: el manager se resuelve como Auth::user()->manager (importer's manager) al momento de confirmar el import.
Digest semanal: el comando agrupa los pendientes (PND businesses + pending_review batches) por manager_id y envía un correo por manager. Si un manager no tiene email registrado, se omite silenciosamente.
